Chapter 4 · Deep Q-Networks

When the table doesn't fit

CartPole's state is four real numbers — there is no row 3.7218 in a Q-table. This chapter replaces the table with a neural network and discovers, the hard way, that Q-learning plus function approximation wants to explode. DQN's two famous stabilizers tame it, and both live in the repo's dqn.py (lineage: CleanRL).

Part 1

From table to function

A Q-table stores one independent number per . That fails twice: continuous states have no rows at all, and huge discrete spaces have too many (nothing learned about one state transfers to its neighbor). The fix is to make Q a parameterized function — a neural network — so that similar states share parameters and therefore share knowledge. That sharing is called generalization, and it's both the superpower and the danger of everything from here to the end of the book.

class QNetwork(nn.Module):
    def __init__(self, env):
        super().__init__()
        self.network = nn.Sequential(
            nn.Linear(np.array(env.single_observation_space.shape).prod(), 120),
            nn.ReLU(),
            nn.Linear(120, 84),
            nn.ReLU(),
            nn.Linear(84, env.single_action_space.n),   # one output PER ACTION
        )

Note the architecture choice: the state goes in, and all action-values come out at once. One forward pass gives you the whole row — so the greedy action and the max in the TD target each cost a single evaluation instead of of them.

Analogy

Memorizing every phone number in the city versus learning how area codes work. A table can only recall what it has seen; a function generalizes — similar states share structure, so one experience teaches many states at once.

In the wild — Atari from pixels

An Atari screen has more possible states than atoms in the universe — no table applies. DQN's convnet maps raw pixels to action-values, and the same architecture learned 49 different games. Generalization is the entire point.

Part 2

Q-learning as regression

The tabular update "move toward the target" becomes a supervised-looking loss — regress the network's prediction onto the one-step bootstrap target:

Two things hide in that formula, and they are the two ideas of this chapter. The target uses parameters — a separate, frozen copy of the network. And the expectation runs over a dataset — a replay buffer, not the last transition. Gradients flow only through the prediction, never the target; this is called a semi-gradient method, because the target is treated as a constant even though it secretly depends on the parameters too.

Analogy

Archery at a target that moves every time you adjust your stance — because the target is computed from your own network. Freezing θ⁻ nails the target to the wall long enough to aim properly.

In the wild — DQN, Nature 2015

Mnih et al. framed Q-learning as exactly this regression and put it on the cover of Nature: one network, one loss, 49 games, human-level on 29. The loss on this page is lifted from that paper.

Part 3

The deadly triad, and DQN's two answers

Sutton & Barto's warning: combine function approximation, bootstrapping, and off-policy training, and convergence guarantees evaporate — values can spiral to infinity. DQN uses all three anyway, and survives on two stabilizers:

Fix 1 — replay buffer

Consecutive transitions are strongly correlated (the pole leans left for thirty frames straight), and SGD hates correlated batches. Store transitions in a FIFO buffer; train on random minibatches from it. Bonus: each transition is reused many times — sample efficiency. This is only legal because Q-learning is off-policy: it doesn't care which stale policy produced the data.

Fix 2 — target network

If the target is computed with the same weights you're updating, every gradient step moves the goalposts — a feedback loop that oscillates or diverges. So the target uses a frozen copy , refreshed only every 500 steps. Between refreshes, DQN solves an honest, stationary regression problem.

Analogy

Replay is shuffling your flashcards instead of studying the textbook in page order — consecutive experiences are too alike to learn from. The target network is grading against last month's answer key, so the goalposts hold still mid-lesson.

In the wild — QT-Opt grasping

Google's QT-Opt trained real robot grasping from a replay buffer of 580,000 stored grasp attempts — off-policy replay is what made reusing months of robot time possible. Both fixes shipped in the original DQN and never left.

Part 4 · from the repo

The update, line by line

This is the heart of dqn.py — the math above, transcribed almost symbol for symbol:

data = rb.sample(args.batch_size)                       # random minibatch from replay
with torch.no_grad():                                   # target: no gradients, ever
    target_max, _ = target_network(data.next_observations).max(dim=1)
    td_target = data.rewards.flatten() + args.gamma * target_max * (1 - data.dones.flatten())

old_val = q_network(data.observations).gather(1, data.actions).squeeze()
loss = F.mse_loss(td_target, old_val)

optimizer.zero_grad()
loss.backward()
optimizer.step()

The (1 - dones) factor is Chapter 3's "zero the bootstrap at terminal states", vectorized. And gather(1, actions) picks, from each row of predicted action-values, the one action that was actually taken — the only entry the loss is allowed to touch. Finally, every 500 steps, the target catches up:

if global_step % args.target_network_frequency == 0:
    for target_param, q_param in zip(target_network.parameters(), q_network.parameters()):
        target_param.data.copy_(args.tau * q_param.data + (1.0 - args.tau) * target_param.data)

With (the repo's default) this is a hard copy; would give a slow exponential blend ("Polyak averaging") — same idea, smoother motion.

Analogy

Following a recipe card: gather ingredients (sample a batch), taste (compute the loss), adjust seasoning (gradient step), and occasionally rewrite your reference notes (sync the target net). Twenty lines, no step skippable.

Part 5

The small print that makes it work

The file's docstring lists what the paper-version leaves out — read these as a checklist for your own implementations:

learning_starts = 10,000: collect experience with no updates first, so the first minibatches are diverse instead of one policy's rut.

train_frequency = 10: one gradient step per ten environment steps — cheaper, and the buffer refreshes between updates.

ε decays linearly from 1.0 to 0.05 over half of training — the same schedule as tabular.py; explore first, exploit later.

The buffer is capped (FIFO, 10k transitions): ancient experience from a terrible early policy eventually ages out.

Adam instead of vanilla SGD — adaptive step sizes forgive the loss surface's changing scale as targets shift.

Want to feel why the stabilizers matter? Project 2: train it, then set target_network_frequency=1 and watch the loss curve panic.

Analogy

A pilot's checklist. The plane flies on aerodynamics, but it lands safely on the boring small print — flap settings, fuel checks. Skip one item and the crash report will blame "implementation details."

In the wild — the Rainbow result

DeepMind's Rainbow paper stacked six such "details" (double Q, prioritized replay, dueling heads, n-step, distributional values, noisy nets) and roughly doubled DQN's Atari score — the small print was worth as much as the idea.
